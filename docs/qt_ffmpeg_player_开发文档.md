# Qt + FFmpeg + C++ 视频播放器开发文档

## 1. 项目目标

本项目要实现一个基于 `Qt + FFmpeg + C++` 的桌面视频播放器，按下面的顺序逐步开发：

1. 先实现“播放无音频视频”
2. 再实现“播放有音频视频”
3. 再加入“倍速播放”
4. 再加入“音量调节”

建议不要一开始就把所有功能堆在一起。播放器最容易出问题的地方不是“能不能解码”，而是：

- 线程模型是否清晰
- 视频帧显示是否稳定
- 音视频同步是否可靠
- 时间戳和缓冲队列是否处理正确

所以本项目应采用“分阶段验收”的方式推进。

## 2. 默认技术方案

本文档先按下面的默认方案设计：

- 操作系统：`Windows 10/11`
- UI 框架：`Qt Widgets`
- 构建方式：`CMake`
- 语言标准：`C++17`
- Qt 版本：`Qt 6.x`
- 编译器：`MSVC 2022 x64`
- 解码库：`FFmpeg`
- 第一阶段渲染方式：`QWidget + QImage`

这样选的原因：

- `Qt Widgets` 上手更直接，适合先把底层播放链路跑通
- `MSVC + Qt 6` 在 Windows 下整体更稳
- 第一阶段先用 `QImage` 显示 RGB 图像，逻辑最清楚
- 等播放链路稳定后，再考虑 `QOpenGLWidget` 做性能优化

如果你后面更希望用 `QML` 或 `MinGW`，文档也可以再改成对应版本。

## 3. 分阶段开发路线

### 阶段 A：环境搭建

目标：

- 能创建并编译一个 Qt 工程
- 能把 FFmpeg 头文件、库文件、DLL 接入工程
- 能成功启动程序并加载本地视频文件

交付结果：

- 空白 Qt 工程可以编译通过
- 程序运行时不报缺少 FFmpeg DLL

### 阶段 B：无音频视频播放

目标：

- 只处理视频流
- 能打开 MP4、MKV 等常见文件
- 能解码视频帧并显示到界面

交付结果：

- 可以播放一个没有音频的视频
- 画面连续，播放顺序正确

### 阶段 C：有音频视频播放

目标：

- 增加音频流解析、解码、重采样和播放
- 实现基础音视频同步

交付结果：

- 可以播放普通带音频的视频
- 声音正常，画面与声音基本同步

### 阶段 D：倍速播放

目标：

- 支持 `0.5x / 1.0x / 1.25x / 1.5x / 2.0x`
- 保证倍速时播放器不明显卡顿

交付结果：

- 无音频视频可倍速播放
- 带音频视频也可倍速播放

### 阶段 E：音量调节

目标：

- 提供 `0 ~ 100` 音量控制
- 支持静音

交付结果：

- 滑块调节可实时生效
- 静音/恢复正常

## 4. 环境安装

## 4.1 推荐工具链

强烈建议你统一使用一套 64 位工具链，不要混用：

- Qt：`MSVC 2022 64-bit`
- 编译器：`MSVC 2022`
- FFmpeg：`x64` 版本

注意：

- `Qt MinGW` 不能直接和 `MSVC` 版库混着用
- `32 位` 和 `64 位` 也不能混用
- Debug/Release 阶段都要注意 DLL 是否能找到

## 4.2 安装 Qt

你已经安装了 Qt，但需要确认安装组件是否完整。

建议在 Qt Maintenance Tool 中确认包含：

- `Qt 6.x MSVC 2022 64-bit`
- `CMake`
- `Ninja`（可选，但推荐）
- `Qt Creator`

如果你装的是 `MinGW` 套件，也能做，但本文档默认按 `MSVC` 说明。

## 4.3 安装 Visual Studio Build Tools

如果你还没装编译器，需要安装：

- `Visual Studio 2022` 或 `Build Tools for Visual Studio 2022`

安装时勾选：

- `Desktop development with C++`
- `MSVC v143`
- `Windows 10/11 SDK`
- `C++ CMake tools for Windows`

安装完成后，确认命令行中可用：

```powershell
cl
cmake --version
```

## 4.4 安装 FFmpeg

Windows 下最重要的不是“下载 FFmpeg 就行”，而是要保证：

- 有头文件 `include`
- 有链接库 `lib`
- 有运行时动态库 `bin/*.dll`

你可以选下面两种方式中的一种。

### 方案 1：手动集成 FFmpeg

在项目目录中建立：

```text
third_party/
  ffmpeg/
    include/
    lib/
    bin/
```

你需要准备这些内容：

- `include`：FFmpeg 头文件
- `lib`：`avcodec.lib`、`avformat.lib`、`avutil.lib`、`swscale.lib`、后期还需要 `swresample.lib`
- `bin`：对应的 `*.dll`

最少先要这些模块：

- `avformat`
- `avcodec`
- `avutil`
- `swscale`

带音频后还要：

- `swresample`

优点：

- 结构直观
- 不依赖外部包管理器

缺点：

- 需要你自己保证库版本、架构、编译器匹配

### 方案 2：用 vcpkg 安装 FFmpeg

如果你愿意把环境管理做得更规范，建议用 `vcpkg`。

典型思路：

1. 安装 `vcpkg`
2. 执行 `vcpkg install ffmpeg:x64-windows`
3. 在 CMake 里接入 vcpkg toolchain

优点：

- 依赖管理更清晰
- 更适合后续扩展

缺点：

- 初学阶段配置量略大

如果你希望，我后面可以单独再给你一份“Windows 下 Qt + vcpkg + FFmpeg 的一步一步安装文档”。

## 4.5 运行时 DLL 部署

程序能编译通过，不代表能运行成功。Windows 下常见问题是启动时报缺少 DLL。

你需要保证：

- Qt 的运行库已部署
- FFmpeg 的 `dll` 已复制到可执行文件目录

Qt 一般可用：

```powershell
windeployqt your_app.exe
```

然后再把 FFmpeg 的这些 DLL 复制到 exe 同目录：

- `avcodec-*.dll`
- `avformat-*.dll`
- `avutil-*.dll`
- `swscale-*.dll`
- 后期再加 `swresample-*.dll`

## 6. 模块设计

## 6.1 核心模块

建议至少拆出以下几个模块。

### 1. `PlayerController`

职责：

- 对外提供 `open / play / pause / stop / seek`
- 协调解封装、解码、渲染、音频输出
- 维护播放器状态

### 2. `DemuxThread`

职责：

- 调用 `av_read_frame`
- 把读取到的 `AVPacket` 分发给视频队列和音频队列

### 3. `PacketQueue`

职责：

- 缓存 `AVPacket`
- 供解码线程消费

要求：

- 线程安全
- 支持阻塞等待
- 支持清空和停止

### 4. `VideoDecoder`

职责：

- 从视频包队列取包
- 调用 `avcodec_send_packet / avcodec_receive_frame`
- 输出视频帧

### 5. `VideoRenderWidget`

职责：

- 把视频帧转换后显示到界面
- 第一阶段可以接收 RGB 图像并绘制

### 6. `AudioDecoder`

第二阶段加入，职责：

- 解码音频帧
- 调用 `swr_convert` 做重采样
- 输出 PCM 数据

### 7. `AudioOutput`

第二阶段加入，职责：

- 使用 Qt 音频接口播放 PCM
- 管理音量
- 给同步模块提供播放时钟

### 8. `AVSyncClock`

第二阶段加入，职责：

- 维护音频时钟和视频时钟
- 计算视频帧应该何时显示

## 6.2 线程模型

建议采用以下线程模型：

- 主线程：界面、用户操作、最终显示
- 解封装线程：读取媒体流
- 视频解码线程：解码视频
- 音频解码线程：第二阶段加入

第一阶段先不要把线程开太复杂。推荐最小化方案：

- 主线程负责 UI
- 一个线程负责解封装
- 一个线程负责视频解码

## 7. 第一阶段：实现无音频视频播放

这是最关键的里程碑。只要这一阶段做对了，后面功能都是增量扩展。

## 7.1 功能范围

第一阶段只做这些事：

- 打开本地视频文件
- 找到视频流
- 初始化视频解码器
- 读取视频包
- 解码成 `AVFrame`
- 转换成 `RGB32`
- 显示到 Qt 控件上

先不要做：

- 音频
- 倍速
- 音量
- 拖动进度条
- 字幕
- 硬件解码

## 7.2 关键 FFmpeg 流程

视频播放主链路：

1. `avformat_open_input`
2. `avformat_find_stream_info`
3. `av_find_best_stream`
4. `avcodec_alloc_context3`
5. `avcodec_parameters_to_context`
6. `avcodec_open2`
7. 循环 `av_read_frame`
8. 把视频包送入解码器
9. `avcodec_send_packet`
10. `avcodec_receive_frame`
11. `sws_scale` 转 RGB
12. 交给 Qt 绘制

## 7.3 视频显示方案

推荐第一版使用：

- FFmpeg 解码输出 `YUV`
- 使用 `sws_scale` 转成 `RGB32`
- 转成 `QImage`
- 在 `QWidget::paintEvent` 中绘制

优点：

- 容易理解
- 容易调试

缺点：

- CPU 占用比 OpenGL 更高
- 高分辨率视频性能一般

但作为第一版足够合适。

## 7.4 第一阶段的时间控制

“能看到画面”不等于“能播放视频”。如果没有时间控制，画面会一下子解码完。

第一阶段可以先采用简化方案：

- 从视频帧的 `pts` 计算显示时间
- 如果 `pts` 不稳定，可先用帧率估算显示间隔
- 每帧到时间后再刷新界面

建议优先使用：

- `best_effort_timestamp`
- 配合流的 `time_base`

显示时间计算思路：

```text
秒 = pts * av_q2d(time_base)
```

第一阶段允许你先做到“基本匀速播放”，后面在音视频同步阶段再细化。

## 7.5 第一阶段验收标准

做到下面几点，就可以进入第二阶段：

- 能打开无音频 MP4 文件
- 能连续显示视频画面
- 关闭播放器不会崩溃
- 连续打开多个文件不会明显泄漏资源
- 停止播放后线程能正常退出

## 7.6 第一阶段最容易踩的坑

### 坑 1：没有处理时间戳

现象：

- 画面一闪而过
- 播放速度明显异常

原因：

- 只解码不按时间显示

### 坑 2：直接跨线程操作 UI

现象：

- 偶发崩溃
- 画面刷新异常

建议：

- 解码线程不要直接操作界面
- 用信号槽或线程安全队列把帧交给主线程

### 坑 3：FFmpeg 资源释放不完整

必须释放：

- `AVFormatContext`
- `AVCodecContext`
- `AVFrame`
- `AVPacket`
- `SwsContext`

### 坑 4：像素格式理解错误

现象：

- 颜色发绿
- 红蓝通道错乱

原因：

- `QImage::Format` 与 `sws_scale` 输出格式不匹配

## 8. 第二阶段：加入音频播放

第二阶段开始处理真正的播放器难点。

## 8.1 新增功能

- 找到音频流
- 初始化音频解码器
- 解码音频帧
- 重采样成 Qt 易播放的 PCM 格式
- 通过 Qt 音频接口播放

## 8.2 推荐方案

推荐使用：

- FFmpeg：负责音频解码
- `swresample`：负责重采样
- Qt：负责音频输出

Qt 6 中可考虑使用：

- `QAudioSink`

思路：

1. 音频帧解码后通常不是最终可播放格式
2. 用 `swr_convert` 转成统一格式，例如：
   - 采样格式：`S16` 或 `Float`
   - 声道布局：立体声
   - 采样率：如 `44100` 或 `48000`
3. 把 PCM 数据喂给 Qt 音频输出

## 8.3 音视频同步策略

建议第二阶段采用最常见、也最稳的方案：

- 以音频为主时钟
- 视频根据音频时间决定显示快慢

原因：

- 音频设备实际播放存在硬件节奏
- 让视频跟随音频，稳定性更高

实现思路：

1. 音频播放时不断更新“当前音频时间”
2. 视频帧到来时比较：

```text
video_pts - audio_clock
```

3. 若视频快了，稍微等一会再显示
4. 若视频慢了，必要时丢帧

第一版同步不需要追求完美，只要做到：

- 肉眼看起来基本同步
- 没有明显积累漂移

## 8.4 第二阶段验收标准

- 能播放带音频的视频
- 声音正常，没有明显爆音
- 声画基本同步
- 暂停、停止时音频设备能正常释放

## 9. 第三阶段：倍速播放

倍速功能分成两种复杂度。

## 9.1 简化版倍速

适用范围：

- 先让“无音频视频”支持倍速

做法：

- 调整视频帧显示间隔

例如：

- `2.0x`：等待时间减半
- `0.5x`：等待时间翻倍

这一版实现很简单，适合先把 UI 和时间控制串起来。

## 9.2 带音频的倍速

这一部分复杂度明显更高，因为必须处理音频。

有两条路线：

### 路线 A：简单实现

做法：

- 通过重采样改变播放速度

特点：

- 实现相对简单
- 声音音调会变化

适合作为第一版“能用即可”的倍速方案。

### 路线 B：高质量实现

做法：

- 引入专门的时域拉伸算法
- 例如后续集成第三方音频时伸库

特点：

- 倍速后音调更自然
- 实现复杂度明显上升

对你这个项目，我建议：

1. 先做路线 A，把功能跑通
2. 后面如果你想把播放器做得更完整，再升级路线 B

## 9.3 倍速功能验收标准

- 无音频视频的倍速播放正常
- 带音频视频在 `0.5x ~ 2.0x` 范围内可播放
- 改变倍速时播放器不会卡死

## 10. 第四阶段：音量调节

音量调节是相对简单的一项功能，但要放在音频链路稳定以后再做。

## 10.1 实现方式

有两种常见方案。

### 方案 A：使用 Qt 音频接口音量属性

如果音频输出接口支持，直接设置音量值。

优点：

- 实现简单
- 代码量少

### 方案 B：软件缩放 PCM 振幅

在把 PCM 数据送到音频设备前，按音量系数缩放采样值。

优点：

- 可控性强

缺点：

- 要处理采样格式细节

对当前项目，优先推荐方案 A。

## 10.2 UI 设计建议

建议加以下控件：

- 打开文件按钮
- 播放/暂停按钮
- 停止按钮
- 倍速下拉框
- 音量滑块
- 静音按钮

## 10.3 音量功能验收标准

- 拖动滑块能实时改变音量
- 音量为 0 时安静无杂音
- 静音/恢复状态正确

## 11. 建议的开发顺序

建议你严格按下面顺序开发，不要跳步：

1. 创建 Qt Widgets + CMake 工程
2. 接入 FFmpeg 头文件、库、DLL
3. 做一个“打开文件”按钮
4. 打通“只解封装视频流”
5. 打通“只解码视频帧”
6. 打通“把视频帧显示到界面”
7. 加入播放节奏控制
8. 清理停止、关闭、资源释放逻辑
9. 再进入音频链路
10. 做音视频同步
11. 做倍速
12. 做音量

这个顺序的目的，是始终保证你每一步都能验证，而不是堆了一大堆代码后不知道哪里错了。

## 12. 第一版最小可用功能清单

如果你要做一个“第一周可交付版本”，建议只包含：

- 打开本地视频文件
- 播放无音频视频
- 停止播放
- 关闭窗口不崩溃

先不要在第一版加入：

- 音频
- 倍速
- 音量
- 进度条拖动
- 全屏
- 截图

## 13. 调试建议

## 13.1 准备测试样本

建议至少准备 4 个测试视频：

- 无音频 MP4
- 有音频 MP4
- 高分辨率视频
- 帧率较低或较高的视频

## 13.2 关键日志

建议打印这些信息：

- 文件是否打开成功
- 找到的视频流索引
- 找到的音频流索引
- 解码器名称
- 视频宽高、帧率、像素格式
- 音频采样率、声道数、采样格式
- 每次播放、暂停、停止、关闭时的状态变化

## 13.3 崩溃排查优先级

如果程序崩溃，先查：

1. 指针是否重复释放
2. 线程退出时对象是否仍被访问
3. `AVFrame` / `AVPacket` 生命周期是否正确
4. `QImage` 是否引用了已失效内存

特别注意：

- 不要让 `QImage` 直接长期引用 FFmpeg 临时缓冲区
- 更稳的做法是复制一份可控数据

## 14. CMake 接入思路

如果你采用手动集成 FFmpeg，CMake 的思路通常是：

1. 指定 Qt
2. 指定 FFmpeg 的 `include`
3. 指定 FFmpeg 的 `lib`
4. 链接以下库：

- `avformat`
- `avcodec`
- `avutil`
- `swscale`
- 第二阶段再加 `swresample`

你当前工程初期建议使用：

- `Qt6::Widgets`

后面做音频输出时，再按需要引入：
